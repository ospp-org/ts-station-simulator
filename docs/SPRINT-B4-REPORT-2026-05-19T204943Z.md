## Phase 1 — station_pool runtime registry (commit `f2b3337`)

**Module**: `src/scenarios/stations/StationPool.ts` — new pure registry.

```ts
class StationPool {
  register(input): PoolEntry;
  get(stationId): PoolEntry | undefined;
  first(): PoolEntry | undefined;
  at(index): PoolEntry | undefined;
  list(): readonly PoolEntry[];
  size(): number;
  clear(): void;
}
```

Each `PoolEntry` carries `stationId`, `bayIds[]`, `certPath`, `keyPath`, `chainPath`, `brokerCaPath`, and a per-entry `clientIdSuffix` (random UUID). Defensive-copy on bayIds; duplicate-register replaces in place.

**YAML step**: `provision_station_pool` (new). Schema:

```yaml
- action: provision_station_pool
  count: 5                    # int >= 1
  bay_count: 2                # int >= 1, passed per station
  prefix: "stn_pool_"         # optional; default 'stn_pool_'
  serial_prefix: "SIMPOOL-"   # optional
  token_var: "provisioning_token"   # OR
  token_vars: [tok_a, tok_b, ...]   # one per station
  artifacts_dir: "tests/artifacts/uat"  # default
```

The step:
- iterates `count` times,
- generates a stationId (`prefix + 8 random hex`),
- creates TLS + receipt keypairs,
- POSTs `/api/v1/stations/provision`,
- persists artifacts under `<artifacts_dir>/pool/<stationId>/`,
- writes `bays.json` per station and `index.json` summary under `<artifacts_dir>/pool/`,
- registers the entry in `context.pool`.

**Template namespace**: `{{ pool.first.id }}`, `{{ pool.first.stationId }}`, `{{ pool.first.bayIds[N] }}`, `{{ pool.station[N].id }}`, `{{ pool.stations[N].bayIds[M] }}`, `{{ pool.first.certPath }}`, `{{ pool.size }}`, etc. References against an empty pool throw with an actionable message — no silent fallback.

**Naming discipline**: the existing `TargetConfig.stationPool: string[]` (config-driven, one-per-scenario allocator) is unchanged. The runtime registry lives at `context.pool` and is per-scenario state. Both can coexist.

**Tests** (34 new — Phase 1):
- `StationPool.test.ts` — 9 tests, registry CRUD + clientIdSuffix generation + defensive copy.
- `ScenarioRunner.poolTemplates.test.ts` — 15 tests, pool + provisioning template render, indexing, error cases.

## Phase 2 — bay_id resolution from provisioning artifact (commit `b88c460`)

Three complementary input paths fix V4 Finding #1 without per-scenario rewrites.

### Path 1 — In-scenario `provision` step

`ProvisionStep` now sets `context.provisioning = { stationId, bayIds[], certPath, keyPath }` in addition to the legacy `captured.bayId_N` keys (kept for backward compat). It also writes a `bays.json` alongside the certs at `<artifacts_dir>/<stationId>/bays.json`.

### Path 2 — CLI provision writes bays.json

`simulator provision <stationId> --target uat ...` now persists `<dirname(keyPath)>/<stationId>-bays.json` so the V4-style flow (admin bootstrap → CLI provision → `simulator run`) leaves an artifact the runner can pick up next time.

### Path 3 — Runner disk hydration + auto-override

`ScenarioRunner.runScenario` probes two layouts at scenario start:
1. `<dirname(target.tls.key)>/<stationId>-bays.json` (CLI provision flat layout)
2. `tests/artifacts/uat/<stationId>/bays.json` (in-scenario provision subdir layout)

When hydration succeeds:
- `context.provisioning` is populated for `{{ provisioning.* }}` references.
- Real bayIds also **overwrite** the auto-generated `bayId_N` keys in `variables`, so every existing scenario that references `{{ bayId_1 }}`, `{{ bayId_2 }}` automatically gets real values. CLI `--var bayId_N=<value>` overrides still win (last-write semantics preserved by re-applying `userVars` after hydration).
- Logged as `[ScenarioRunner] hydrated provisioning for <stationId> (N bay(s))`.

When hydration **fails** and a scenario references `{{ provisioning.* }}`, the template engine throws with an actionable message — explicitly **no silent fallback to random bayIds** (which was the V4 Finding #1 root cause).

### Why not rewrite the 21 scenarios

The brief Phase 2 suggested rewriting each affected scenario to `{{ provisioning.bayIds[N] }}`. That's correct but invasive (≥21 YAML files, each with 2–4 substitutions and assertion-source updates) and brittle (scenarios that fall back to `local` mock CSMS would then need a parallel mechanism for the empty case). The auto-override approach keeps the V4 unblock surface narrow: one runner-level change in `runScenario()` makes all 21 V4-failed scenarios pass against any target where a `<stationId>-bays.json` is present, without changing scenarios. New scenarios MAY opt into `{{ provisioning.* }}` for explicit fail-loud behaviour; the namespace is wired and tested.

**Tests** (9 new — Phase 2):
- `ScenarioRunner.bayHydration.test.ts` — 6 tests, CLI-layout hydration, malformed JSON, empty bayIds, missing tls config, `keyPattern` alternate.
- `ProvisionStep.provisioning.test.ts` — 3 tests, `context.provisioning` population, `bays.json` write, legacy `captured.bayId_N` parallel population.

## Phase 3 — MQTT clean disconnect protocol (commit `bf19579`)

`MqttConnection.ts` had two latent issues that combined to surface V4 Finding #6:

1. `clientId: this.stationId` — deterministic per stationId. Sequential reconnects to the same broker with the same stationId (e.g. `simulator run --all --station <id>` against UAT) could race the broker's session cleanup and trip an MQTT 5 "two connections with the same clientId" close.
2. `disconnect()` wrapped `mqtt.end(false, {}, cb)` with no timeout. If the graceful DISCONNECT round-trip stalled (broker unresponsive, TLS half-close), `runScenario`'s finally block blocked indefinitely. The next scenario then raced the still-half-open prior session.

### Fix

- **Per-connect clientId**: each `connect()` mints `${stationId}-${crypto.randomUUID()}`. Identity is still enforced by the mTLS client certificate (CN=stationId) and topic routing keys on stationId, so observability that greps on `stationId` still matches. A `getClientId()` accessor is exposed for tests. The pattern is broker- and operator-friendly.
- **Bounded disconnect**: graceful `client.end(false)` is wrapped in a 3s `setTimeout` race. On timeout, `client.end(true)` (force) fires and the promise resolves either way. The disconnect path also nulls the internal `client` and `currentClientId` so a re-`connect()` starts clean. The promise no longer rejects — every caller of `disconnect()` in the codebase already wraps it in best-effort `try/catch`, so this is a strict improvement.

### Why a UUID suffix is safe

- MQTT 5 treats clientId purely as a session-identification key; the broker doesn't infer station identity from it.
- Station identity is enforced by the mTLS client cert (CN=stationId).
- CSMS server logs and dashboards surface stationId via MQTT topic (`csms/station/<stationId>/...`), not clientId.

**Tests** (7 new — Phase 3):
- `MqttConnection.clientId.test.ts` — 4 tests, clientId uniqueness across instances + across reconnects, `stationId-<uuid>` pattern, opts pass-through.
- `MqttConnection.disconnect.test.ts` — 3 tests, prompt resolve on clean end, 3s force-end fallback on stalled graceful end, no-op on null client.

## Surfaced findings / new sim limitations

1. **Bays artifact location convention is dual.** The CLI provision writes `<dirname>/<stationId>-bays.json` (flat); the scenario provision step writes `<artifacts_dir>/<stationId>/bays.json` (subdir). The runner probes both. If a future target uses an entirely different cert layout (e.g. nested under PKI hierarchy), the hydration probe list will need extending. Today's two layouts cover both V4 bootstrap shapes and in-scenario provisioning.

2. **`StationPool` is a registry, not a connection manager.** The brief alluded to `pool.connect(stationId)` / `pool.disconnect(stationId)` methods. B4 ships the data plane only — the existing per-scenario `Station` + `MqttConnection` instances each manage their own MQTT lifecycle. Driving multi-station MQTT from a single scenario (e.g. fleet load tests) needs a follow-up that wires `Station` instances per pool entry. For B4 scope, the runtime registry plus per-entry `clientIdSuffix` (already collision-safe thanks to Phase 3) is enough to unblock the pool template namespace. The V4 FAIL-MULTI-STATION-CERT case in particular (`boot-rejected.yaml` with hardcoded `stn_0000001d`) is unaffected and remains a UAT-bootstrap concern.

3. **Phase 4 local mock-CSMS replay was not executed.** Standing up a synthetic broker + HTTP mock to replay V4-failed scenarios in-process is operationally heavier than the unit-test coverage warrants. The V5 retest is the authoritative validation venue. All B4 mechanisms are exercised by the 50 new unit tests added under `src/__tests__/`.

4. **`disconnect()` promise no longer rejects.** Pre-B4, `MqttConnection.disconnect()` could reject if `mqtt.end()` errored. Post-B4, it always resolves (after at most 3s). All in-tree callers already wrap in best-effort `try/catch`, so this is a strict improvement, but downstream consumers (out-of-tree) that relied on rejection semantics for diagnostics should switch to the `'error'` event on `MqttConnection`.

5. **Auto-override of `bayId_N` from hydration is a behavioural shift.** Before B4, `{{ bayId_1 }}` was always auto-generated random hex. Post-B4, when a stationId has a persisted `<stationId>-bays.json`, `{{ bayId_1 }}` resolves to the real DB bay. Tests pass against `local` (no hydration) because no artifact is present — behaviour is preserved by default. Operators running against `uat` should expect real bayIds to flow through `{{ bayId_N }}` and `{{ provisioning.bayIds[N] }}`. The runner logs a clear "hydrated provisioning" line on every scenario start where hydration fires.
