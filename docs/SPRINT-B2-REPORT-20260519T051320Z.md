## 1. Phase 1 — `--var` on `simulator connect`

### 1.1 Usage

```bash
npx simulator connect \
  --target uat \
  --station stn_realstation \
  --var bayId_1=bay_realbay001 \
  --var bayId_2=bay_realbay002
```

The flag is repeatable. Each `KEY=VALUE` pair lands in a `Map<string, string>` and `bayId_<N>` keys override the deterministic bayId in slot N. Keys outside that shape are still parsed (validated by the existing `parseUserVars` validator) but emit a warning since connect mode only honors bayId overrides today.

Smoke run (real output):

```
$ npx tsx src/cli/index.ts connect --target local --station stn_test1234 --var bayId_1=bay_realbay001
Connecting station stn_test1234 to local...
  Overrides (1): bayId_1=bay_realbay001
[Station] Retrying BootNotification...

Station stn_test1234 connected. BootNotification sent.

Station IDs:
  stationId:  stn_test1234
  bay 1:      bay_realbay001  (service: svc_wash_basic)
  bay 2:      bay_test123402  (service: svc_wash_basic)
```

Warnings for out-of-range / unrelated keys:

```
$ npx tsx src/cli/index.ts connect --target local --station stn_test1234 --var bayId_3=bay_oor --var foo=bar
Connecting station stn_test1234 to local...
  Warning: --var bayId_3=... out of range (bayCount=2, requested bay #3; ignored)
  Warning: --var foo=... not recognized by connect mode (only bayId_<N> is honored; ignored)
  Overrides (2): bayId_3=bay_oor, foo=bar
```

### 1.2 Code paths touched

| File | Change |
|---|---|
| `src/cli/connectBays.ts` (NEW) | Pure helper `deriveBays(stationId, bayCount, userVars) → { bays, warnings }` |
| `src/cli/index.ts` | Added `--var <pair>` option to `connect`, parse + log + apply via `deriveBays`. `ConnectCommandOptions.var: string[]`. |
| `src/__tests__/cli/connect-vars.test.ts` (NEW) | 12 unit tests for `deriveBays` |

Validation is unchanged — the existing `parseUserVars` regex (`KEY = /^[A-Za-z_][A-Za-z0-9_]*$/`, `VALUE = /^[A-Za-z0-9_-]+$/`) covers connect mode without modification.

### 1.3 Sub-task — artifact bayIds persistence (NOT shipped this sprint)

The brief offered an alternative path: persist provisioning-response `bayIds` into `certs/<target>/<station>-mqtt.json` so connect mode can read them as defaults. Provisioning already CAPTURES `data.bayIds` from the spec-v0.3.0 response (`src/cli/index.ts:644-646` prints them) but `persistBrokerArtifacts` only writes `brokerRootCa` + `mqttConfig` — bayIds aren't persisted.

**Decision**: not pursued this sprint. `--var` alone satisfies the primary deliverable. Rationale: the artifact-persistence path expands provisioning's surface area (new schema field on the persisted JSON, new load path, plus updates to `artifacts.test.ts`) for an enhancement that's not blocking csms-server Sprint V4 coverage. Future-work bullet captured in §7.
