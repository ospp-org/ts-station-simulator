## Scope re-evaluation surfaced during research

The brief assumed a uniform path rewrite: 39 api_call sites currently calling the non-existent `/api/v1/stations/{stationId}/trigger-command` should all be re-pointed at the new `/api/v1/testing/trigger-command/{stationId}` endpoint.

**Reality check** (from cross-repo Sprint A3 contract extraction):

The new testing endpoint accepts exactly **5 server→station commands**: `GetConfiguration`, `ChangeConfiguration`, `Reset`, `SetMaintenance`, `TriggerMessage`. Any other `command` value returns 422.

The 39 api_call sites use **6 distinct command shapes**:

| Command | Count | Eligible for testing endpoint? | New target |
|---------|-------|-------------------------------|------------|
| StartService | 23 | No (customer-side action) | `POST /api/v1/sessions/start` |
| ReserveBay | 8 | No (customer-side action) | `POST /api/v1/reservations` |
| StopService | 5 | No (customer-side action) | `POST /api/v1/sessions/{id}/stop` |
| CancelReservation | 1 | No (customer-side action) | `POST /api/v1/reservations/{id}/cancel` |
| Reset | 1 | Yes | `POST /api/v1/testing/trigger-command/{stationId}` |
| ChangeConfiguration | 1 | Yes | `POST /api/v1/testing/trigger-command/{stationId}` |
| **Total** | **39** | | |

Only **2 of 39** sites were eligible for the testing endpoint. The remaining 37 are customer-side actions that map to the production `/api/v1/sessions/*` and `/api/v1/reservations/*` routes. A research sub-agent confirmed these routes exist and are fully wired in csms-server — no upstream blocker (Sprint A4 not required for B3 completion).

The rewrite was executed as a **2-endpoint-family bifurcation** with per-command body shape, capture, expect_status, and `background:` flag changes.

## Per-command transformation summary

### StartService (23 sites) → POST /api/v1/sessions/start

- URL: `…/stations/{id}/trigger-command` → `…/sessions/start`
- Body: `{action: "StartService", payload: {sessionId, bayId, serviceId, durationSeconds, sessionSource, reservationId?}}` → `{bay_id, service_id, duration_seconds, reservation_id?}` (snake_case; drop hardcoded sessionId and sessionSource)
- `expect_status`: 200 → **201** (accepted) or **422** (rejected paths: bay-busy, faulted, invalid-service, maintenance)
- **`background: true` added** — POST /sessions/start blocks on SessionReadySignal::wait pending the station's MQTT StartService Response. Without it, scenario deadlocks (fetch awaits HTTP, but server needs simulator's MQTT first).

### StopService (5 sites) → POST /api/v1/sessions/{captured.sessionId}/stop

- URL: `…/stations/{id}/trigger-command` → `…/sessions/{captured.sessionId}/stop`
- Body: removed entirely (empty body; sessionId moves to URL path)
- `expect_status`: 200 → 200
- No `background: true` — stop is server-side non-blocking
- **Special case**: `stop-service-rejected.yaml` — the original tested station-side rejection (errorCode 3006) via thin pass-through. New endpoint pre-validates sessionId server-side and returns 404 before MQTT dispatch, so the wait_for+Rejected MQTT loop is unreachable. Scenario rewritten to test 404 server-side; description updated; original station-rejection coverage flagged as Sprint B3 follow-up.

### ReserveBay (8 sites) → POST /api/v1/reservations

- URL: `…/stations/{id}/trigger-command` → `…/reservations`
- Body: `{action: "ReserveBay", payload: {bayId, reservationId, expirationTime, sessionSource}}` → `{bay_id, duration_minutes, session_source}` (snake_case; drop hardcoded reservationId; expirationTime → duration_minutes derived value)
- `expect_status`: 200 → **201** (accepted) or **422** (rejected)
- No `background: true`

### CancelReservation (1 site) → POST /api/v1/reservations/{captured.reservationId}/cancel

- URL: path param, body empty
- `expect_status`: 200 → 200

### Reset (1 site) → POST /api/v1/testing/trigger-command/{stationId}

- URL: `…/stations/{id}/trigger-command` → `…/testing/trigger-command/{id}` (id moves to end)
- Body: `{action: "Reset", payload: {type}}` → `{command: "Reset", payload: {type}}` (rename `action` → `command`)
- `expect_status`: 200 → **202** (async dispatch enqueued)

### ChangeConfiguration (1 site) → POST /api/v1/testing/trigger-command/{stationId}

- URL: same testing endpoint
- Body: `{action: "ChangeConfiguration", payload: {keys: [{key, value}]}}` → `{command: "ChangeConfiguration", payload: {key, value}}` (flatten keys array to single key/value pair — endpoint takes one at a time)
- `expect_status`: 200 → **202**

## Header strategy chosen

**No YAML header wiring required.** ScenarioRunner already auto-handles both authentication headers natively in `src/scenarios/steps/ApiCallStep.ts`:

- `Authorization: Bearer <jwt>` — `ensureAuth()` lazy-logs in via `POST /api/v1/auth/login` with `context.apiCredentials` (sourced from `config/targets.yaml` per-target). Token cached per `(baseUrl, email)` in module-scope `tokenCache` for parallel-run reuse. Injected at line 223.
- `X-Idempotency-Key: <uuid>` — auto-injected for all POST/PUT/PATCH at line 225 via `methodRequiresIdempotencyKey()` regardless of URL.

Rejected alternatives:
- **Option 1 (auto-JWT via `generateVariables()` scope)** — unnecessary; ensureAuth already covers this implicitly.
- **Option 2 (`--var auth_jwt=…` on CLI)** — unviable; `src/cli/userVars.ts:2` regex `^[A-Za-z0-9_-]+$` rejects JWT `.` separators.

`X-Organization-Id` is auto-injected only for `/api/v1/admin/*` URLs (regex `ApiCallStep.ts:184`). The new URL families don't match:
- `/api/v1/sessions/*` and `/api/v1/reservations/*` — production routes carry only `auth.jwt` + `idempotency` middleware (no `org.context`). JWT alone suffices.
- `/api/v1/testing/trigger-command/*` — carries `org.context` but auto-resolves when the JWT user has a single org membership (typical UAT test user setup).

If multi-org test users are introduced later, `--org-id <uuid>` CLI flag is the explicit escape hatch (`src/cli/index.ts:85`). No YAML changes needed today.
