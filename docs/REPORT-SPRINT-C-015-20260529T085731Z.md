## 1. TL;DR

C-015: `scenarios/security/offline-pass-authorize.yaml` failed because the sim sent `signature: "test_sig"`. The SPEC GATE revealed the original premise ("sim renders the signature") is **impossible** under OSPP — offline passes are signed by the **server** (the issuer) and validated against the **server's own** key; there is no user/sub private key. The correct, zero-server-change fix is **issue-then-present**: the scenario obtains a genuinely server-signed pass from `POST /api/v1/offline/passes` and presents it verbatim via `AuthorizeOfflinePass` — exactly what a real mobile app does.

Delivering that required a minimal, backward-compatible **engine** change (typed whole-value capture injection) plus a symmetric **linter** change. Both are implemented and unit-tested (235/235 green, build clean, scenario lints clean).

**Live validation could not be performed** because the UAT environment is currently degraded in two independent ways (see §6) that are C-018 / environment-bootstrap concerns, explicitly out of C-015 scope, and not fixable from this session (UAT is remote; no local UAT container; no remote DB access).

## 2. SPEC GATE verdict — PASS (format), with a corrected premise

Format/canonical/encoding are **unambiguous**:
- `offline-pass.schema.json:119-124` — `signature`: base64, `maxLength 1024`, "ECDSA P-256 base64-encoded signature over the pass fields"; `signatureAlgorithm` const `ECDSA-P256-SHA256` (`:114-117`).
- `profiles/offline/offline-pass.md:42-50` — data-to-sign = all pass fields **except** `signature`/`signatureAlgorithm`, serialized via **OSPP Canonical Form** (`06-security.md §4.8 :570-621` — recursive lexicographic key sort, compact, UTF-8), SHA-256, ECDSA-P256, **DER → base64**.
- Server cross-check confirms exact match: `EcdsaService.php:88-101` (strip sig fields → canonical serialize → verify), `:50-71` (`base64_decode` strict → `openssl_verify` SHA-256, DER), `CanonicalJsonSerializer.php:7-26`.

**Corrected premise (the decisive finding):** `profiles/offline/offline-pass.md:42` — *"The server signs the OfflinePass"*; the station only verifies (`:49-51`). The server validates `AuthorizeOfflinePass` against **its own public key** (`PassValidator.php:54-55`). So the sim cannot produce a UAT-accepted signature by signing locally — it does not (and must not) hold the server's signing key. `"test_sig"` fails at `base64_decode('test_sig', true) === false` (underscore not in base64 alphabet) before any crypto.

**Decision (Gabi):** Opt 1 — issue-then-present via the issuance API. A server-issued pass is signed with the same key the validator checks → **guaranteed to verify, in any environment**.

## 3. Why a TS change was required (and approved)

Faithfully forwarding a server-signed pass requires byte/type fidelity (the server re-canonicalizes and re-verifies). The template engine **coerced every `{{captured.X}}` to a string** (`ScenarioRunner.ts:270`), and the runner pre-substitutes the whole step at `:663` — so:
- `offlinePass: "{{captured.offlinePass}}"` → `String(object)` = `"[object Object]"` (broken JSON);
- field-by-field rebuild → integers become strings (`policyVersion: "1"` ≠ `1`) → canonical form differs → signature fails.

Approved fix (Decision 2): **typed whole-value capture injection** — when a field's *entire* value is a single `{{captured.X}}` token, return the captured value with its type intact.

**6a. All pool stations unbootable.** Every `station_pool` station (`stn_e38cd38de471` … `stn_c6e08ee9fbb3`) returns `BootNotification → Rejected`. **6b. Offline issuance forbidden for the test users.** `POST /api/v1/offline/passes` returns `403 ACTION_NOT_PERMITTED` `ospp_code 2008` — *"Offline mode is not enabled for this user"* — for **both** the funded customer `e2e-stn_e38cd38de471@onestoppay.dev` and the platform admin `e2e-admin@onestoppay.ro`. Source: `IssueOfflinePassAction.php:61-66` requires `userQuery->isOfflineEnabled()` (i.e. `users.offline_enabled = true`); both users currently have it **false**.

**To validate C-015 live, the UAT env needs (C-018 / bootstrap work):**
1. A **registered + active** station (re-register/activate a pool station, or provision a fresh one), and
2. The issuing user set **`offline_enabled = true`** with a non-negative wallet balance (the customer already has balance 1,000,000).

Neither is fixable from this session: UAT is remote, there is no local UAT app/DB container (`csms-app` here is `APP_ENV=local`), and no admin endpoint to flip `offline_enabled` was found. This is the same class of work Sprint 3 explicitly STOPPED on (C-018, `SPRINT-3-REPORT §8.3`).
